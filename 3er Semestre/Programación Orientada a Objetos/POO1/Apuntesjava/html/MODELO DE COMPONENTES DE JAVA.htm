MODELO DE COMPONENTES DE JAVA

 

El modelo de componentes

Los componentes son elementos autocontenidos que pueden ser controlados dinámicamente y ensamblados para formar aplicaciones. Los componentes deben interoperar de acuerdo a un conjunto de reglas. Deben comportarse de la forma esperada.

 

JAVA BEANS

Definición

Un Bean es un componente de software reutilizable que puede ser manipulado visualmente en una herramienta constructora.

 

Otra definición

Los JavaBeans son el modelo de componentes de Java.

 

El modelo de componentes de Java permite a los usuarios construir aplicaciones reuniendo componentes visualmente o mediante programación. El soporte a la programación visual es muy importante para el modelo de componentes; esto es lo que hace el desarrollo de software basado en componentes verdaderamente poderoso.

 

Descubrimiento y registro

Descubrimiento de clase y interfaz es el mecanismo usado para localizar un componente en tiempo de ejecución y para determinar las interfaces que soporta de modo que esas interfaces puedan ser usadas por otros.

 

Manejo y lanzamiento de eventos

Un evento es algo de importancia que pasa en un momento especifico. Los componentes enviaran notificaciones a otros objetos cuando tenga lugar un evento en el cual esos objetos hayan expresado interés (es decir se hayan registrado como listeners(escuchas)).

 

Persistencia

Ya se vio anteriormente que la persistencia conserva el estado de un objeto en el tiempo y el espacio.

 

Generalmente todos los componentes tienen estado. Dicho estado puede ser almacenado en disco y recuperado de disco. Los componentes deben ser capaces de participar en el mecanismo de persistencia de su contenedor de forma tal que todos los componentes en la aplicación puedan proveer persistencia en una manera uniforme.

 

Soporte para la programación visual

Los componentes son representados en cajas de herramientas. El usuario puede elegir un componente de la caja de herramientas y colocarlo en un contenedor, eligiendo su tamaño y posición. Las propiedades del componente pueden ser editadas para crear el comportamiento deseado. Esto se logra a través de un conjunto de interfaces provistas por el ambiente del componente que permiten a los componentes publicar o exponer sus propiedades.

La arquitectura JavaBeans

Otra definición

JavaBeans es una arquitectura para usar y construir componentes en Java.

 

Los Beans no necesariamente tienen una representación visual en tiempo de ejecución. Pero deben permitir que los valores de las propiedades sean cambiados a través de algún tipo de interfaz visual, y sus métodos y eventos deben ser expuestos para que la herramienta de desarrollo pueda escribir código capaz de manipular el componente cuando se ejecuta la aplicación.

 

Ejemplo de un Bean muy simple

 

public class MiBean implements java.io.Serializable

{

                protected int elValor;

               

                public MiBean(){}

                public void setMiValor(int nuevoValor){ elValor=nuevoValor; }

                public int getMiValor(){ return elValor; }

}

 

Este es un Bean real llamado MiBean que tiene estado (la variable elValor) que será automáticamente almacenada y recuperada por el mecanismo de persistencia de los JavaBeans (para esto MiBean declara que implementa java.io.Serializable), y tiene una propiedad llamada MiValor que es utilizable por los ambientes de programación visual.

Propiedades, Métodos y Eventos

Las propiedades son atributos de un Bean que son referenciados por nombre. Esas propiedades son usualmente leídas y escritas llamando métodos en el Bean creados específicamente para ese propósito. Un programador puede obtener o establecer el valor de una propiedad a través de llamadas a métodos, mientras que un desarrollador de aplicaciones que use una herramienta de desarrollo visual manipulara el valor de la propiedad usando un editor visual de propiedades.

 

Los métodos de un Bean son los métodos Java expuestos por la clase que implementa el Bean. Aquellos métodos representan la interfaz usada para acceder y manipular el componente.

 

Los eventos son el mecanismo usado por un componente para enviar notificaciones a otro. Siempre que un evento ocurre, el componente interesado será notificado mediante la invocación de uno de sus métodos.

 

Ejemplo de un Bean que tiene: Propiedades, Métodos y Eventos

 

package tlight;

import java.awt.*;

import java.awt.event.*;

import java.awt.image.*;

import java.beans.*;

 

//Un Bean simple que representa un componente de luz de trafico

public class TrafficLight extends Canvas implements Runnable {

  private ActionListener l = null;

  private Thread me = null;

  private String defaultLight = "ROJO";

  private String currLight = defaultLight;

 

  private boolean debug = false;

  //Establecer el tamaño del despliegue del Bean

  public TrafficLight () {

    setSize (100, 200);

  }

  // Encender/Apagar la depuración

  public void setDebugging (boolean state) {

    debug = state;

  }

 

  public boolean getDebugging () {

    return debug;

  }

  // Methods for get/set of the default light state

  public synchronized String getDefaultLightState () {

    return defaultLight;

  }

 

  public void setDefaultLightState (String state) {

    defaultLight = state;

    currLight = state;

    repaint();

  }

  // Desplegar el estado actual de la luz de trafico

  public void paint (Graphics g) {

    // Poner un borde negro alrededor de este

    g.setColor (Color.black);

    g.drawRect (1, 1, 98, 198);

    //Pintar el contorno de la luz trafico

    g.setColor (Color.white);

    g.fillRect (2, 2, 96, 196);

    //Pintar el contorno de las luces

    g.setColor (Color.black);

    g.drawOval (30, 30, 40, 40);

    g.drawOval (30, 80, 40, 40);

    g.drawOval (30, 130, 40, 40);

    //Depurar

    if (debug) {

       System.out.println

          ("El estado actual de la luz es: " + currLight);

    }

    //Cual luz esta encendida?

    if (currLight.equals("ROJO")) {

      g.setColor (Color.red);

      g.fillOval (30, 30, 40, 40);     

    } else {

      if (currLight.equals("AMARILLO")) {

        g.setColor (Color.yellow);

        g.fillOval (30, 80, 40, 40);

      } else   {

        if (currLight.equals("VERDE")) {

          g.setColor (Color.green);

          g.fillOval (30, 130, 40, 40);

        }

      }

    }

  }

  //Enviar un mensaje a la luz para empezar un ciclo

  public void start (ActionEvent x) {

    //Depurar

    if (debug) {

      System.out.println ("Obtuve un evento start!");

    }

    startCycle ();

  }

  //Empezar un ciclo de luz

  private synchronized void startCycle () {

    //No molestarse a menos que no haya un ciclo ejecutándose ya

    if (me == null) {

      me = new Thread (this);

      me.start ();

    }

  }

 

  //El método run

  public void run () {

    //Depurar

    if (debug) {

      System.out.println ("Ciclo empezado");

    }

    while (me != null) {

      try {

         me.sleep (3000);

       } catch (InterruptedException e) {

      }

      //Obtener el estado actual de la luz

      if (currLight.equals ("ROJO")) {

        // Cycle to green

        currLight = "VERDE";

        fireAction(currLight);

      } else {

        if (currLight.equals("VERDE")) {

          // Ciclo a través de amarillo a rojo

          currLight = "AMARILLO";

        } else {

          //De otro modo, estamos en AMARILLO,

          //Así ciclo a ROJO

          currLight = "ROJO";

          fireAction(currLight);

        }

      }

      repaint ();

      //Romper el ciclo cuando el estado por default sea alcanzado

      if (currLight == defaultLight) {

        me = null;

      }

    }

  }

  //Obtener un evento que indica un cambio de luz

  //de otra luz

  //En which caso, actuaremos como esclavo

  public void lightChange (ActionEvent x) {

    //Verificar el estado de este evento

    String command = x.getActionCommand();

    // Debug

    if (debug) {

      System.out.println ("Recibí un evento de luz de trafico: "

        + defaultLight + " command: ir a " + command);

    }

    //Si la otra luz cambio a rojo, entonces verificar nuestro default

    if (command.equals ("ROJO")) {

      if (currLight.equals ("VERDE")) {

        //Bien, no hacer nada

      } else {

        // Cycle

        currLight = "VERDE";

        repaint ();

      }

    } else {

      if (command.equals ("VERDE")) {

        if (currLight.equals ("ROJO")) {

          // Bien, no hacer nada

        } else {

          //Ciclo

          currLight = "AMARILLO";

          repaint ();

          try {

            Thread.sleep (3000);

          } catch (InterruptedException e) {

          }

          currLight = "ROJO";

          repaint ();

        }

      }

    }

  }

  //El metodo fireAction envía un evento al

  //TrafficLight esclavo para decirle estamos cambiando de

  //estado

  public void fireAction (String s) {

    //Depurar

    if (debug) {

      System.out.println ("Generando un evento action");

    }

    if (l != null) {

      ActionEvent actionEvt = new ActionEvent(this, 0, s);

      l.actionPerformed(actionEvt);

    }

  }

  //Registrarnos nosotros mismos como la fuente de un evento

  //Solo use Action por ahora...

  public void addActionListener (ActionListener l)

      throws java.util.TooManyListenersException {

    //Depurar

    if (debug) {

      System.out.println ("Registrando a un escucha");

    }

    //Hay ya un escucha de eventos?

    if (this.l != null) {

      throw new java.util.TooManyListenersException ();

    }

    this.l = l;

  }

 

  public void removeActionListener (ActionListener l) {

    l = null;

  }

}

 

Introspección

Introspección es el proceso de exponer las propiedades, los métodos y eventos que soporta un componente JavaBean. Este proceso es usado en tiempo de ejecución, así como por una herramienta de desarrollo visual en tiempo de diseño. El comportamiento por default de este proceso permite la introspección automática de cualquier Bean. Un mecanismo de reflexión de bajo nivel es usado para analizar la clase de un Bean para determinar sus métodos. Después aplica algunos patrones de diseño simples para determinar las propiedades y eventos que son soportados. Para sacar partido de la reflexión, solo se necesita seguir un estilo de codificación que concuerde con el patrón de diseño.

 

También se puede elegir implementar una clase BeanInfo la cual provee información descriptiva acerca de su Bean asociado explícitamente.
